> Identity: **Adversarial-Agent**

## 1. The mandate, restated

Goal: collect **internet-scale, license-clean, world-model-useful video
data**, with cooking videos as the current vertical. No payments, no ToS
violations, no DMCA-anti-circumvention exposure. Be systematic, robust, and
pick frontier methods, not folkloric ones.

This document is the day-one adversarial review and plan.

## 2. Adversarial review of the obvious approach (and why we reject it)

The "obvious" 2026 approach to bootstrap a cooking-video corpus is some
variant of:

1. Take a YouTube ID list (HowTo100M / HD-VILA-100M / Panda-70M / InternVid)
2. Run `yt-dlp` over it
3. Cut into clips, run a VLM captioner, ship.

I am rejecting this approach as the *primary* pipeline. Reasons:

- **Active class-action litigation in 2026.** *Chmura v. Snap*, and the
  YouTube-creator class actions filed against Amazon, OpenAI, and Apple in
  April 2026, all hinge on the theory that **using `yt-dlp` (or equivalent)
  to acquire YouTube videos at scale is DMCA-§1201 anti-circumvention** of
  YouTube's TPMs, regardless of whether the underlying clips are tagged
  Creative Commons. *(See `docs/artifacts/legal-landscape.md`.)*
- **YouTube's ToS overrides CC tags on YouTube.** The V3C dataset
  authors picked Vimeo over YouTube *for exactly this reason*. CC license
  permits redistribution of the work; the platform's ToS still binds the
  acquirer.
- **HD-VILA-100M and Panda-70M licenses say "academic use only."** Using
  them to build *any* commercializable artifact, even an internal one,
  invites the same complaint pattern named in *Chmura*.
- **HowTo100M provides only `youtube_id` + timestamps.** Resolving those
  IDs to media bytes still requires a YouTube acquisition step.

So we treat YouTube as **out of scope for byte-level acquisition**, and
treat YouTube-derived ID lists as **stale-only metadata signals** (e.g.
"this WikiHow task has lots of demonstration videos somewhere on the open
web; go look for license-clean substitutes").

## 3. The license-clean substitute graph

Sources we *can* legally crawl and (in most cases) redistribute clips
from. Each line: source — license posture — coverage notes.

| Source | License posture | Cooking-relevant volume (rough) |
|---|---|---|
| **Wikimedia Commons** | All files are CC-BY/CC-BY-SA/CC0/PD by policy | small but verifiably clean (~75 cooking videos in the canonical category, plus broader food/kitchen categories) |
| **Internet Archive — Prelinger / Community Video** | PD or CC; per-item check required | hundreds of hours of mid-century home-economics films, modern home-cook uploads |
| **Vimeo CC search** | CC tags are platform-honored *and* Vimeo ToS allows download of CC content | unknown, indexable via Vimeo API |
| **Flickr Public Domain / CC video** | per-item license, downloads permitted by ToS for CC items | small but real |
| **Government / public-broadcaster open archives** | varies; e.g. USDA Food Safety, NHS, ABC Australia open content, BBC R&D non-commercial | mid |
| **Creative Commons Search (CC Search / OpenVerse)** | aggregator across the above | important entry point |
| **LibriVox-style "open recipe demo" community uploads on PeerTube** | per-instance, mostly CC | small but growing |
| **Common Crawl WAT/WARC for `<video>` tag URLs** | metadata only — points us to media we then re-license-check | used as a *discovery* layer, not an acquisition layer |

This is the **substitute graph**: for any "this would be great in cooking
data" idea sourced from a YouTube-shaped paper, we resolve it to one of
these instead, or skip it.

## 4. What "frontier" means for video data collection in 2026

The 2026 frontier — distilled from DreamDojo (NVIDIA, 44k egocentric
hours), Vid2World, the a16z "Frontier Systems for the Physical World"
note, and the World-Action Models (WAM) line — has shifted from
*captioned-video pretraining* toward **action-conditioned, temporally
coherent, physically plausible video as a prior for embodied agents.**

Key implications for our flywheel:

- **Per-clip captions are no longer enough.** We want richer signals:
  shot boundary, motion magnitude, hand-presence, tool-presence, physical
  contact events. These are cheap to compute and dramatically increase
  downstream WAM utility per byte. Summer-22B's "multi-stage filtering"
  recipe (TransNet shot detection, DOVER aesthetics, optical flow) is the
  reference architecture.
- **Aesthetic quality is correlated with, but not equal to,
  world-model utility.** A static, beautifully-lit montage of plated food
  is a *bad* WAM example; a shaky 30-second clip of a knife slicing an
  onion is a *good* one. We need to disentangle these.
- **Egocentric and "from-the-hands" is gold.** EPIC-KITCHENS-100 (100h),
  HD-EPIC (CVPR'25), Ego4D, and DreamDojo's egocentric set demonstrate
  this. The substitute graph above is mostly third-person; closing this
  gap is a 2026 research question for us (see §7).

## 5. The proposed pipeline (v0)

Single source-of-truth: `db_structured.md`. Each stage emits a JSON Lines
manifest with provenance.

```
discover → license_check → segment → score → dedup → curate → ship
   │            │            │         │       │        │
   ▼            ▼            ▼         ▼       ▼        ▼
 raw_video   raw_video    clip      clip   clip       shard
 .jsonl      .jsonl       .jsonl    .jsonl .jsonl     .jsonl
                                    +features
```

- **discover**: per-source connector; emits `raw_video` rows with at minimum
  URL, license tag, source, attribution. *No bytes downloaded yet.*
- **license_check**: rejects rows without an SPDX-mappable license.
- **segment**: PySceneDetect for v0; TransNet-v2 for v1. Records
  `start_s`/`end_s`/`splitter`.
- **score**: pHash (DCT, 64-bit), motion (optical flow magnitude), DOVER
  for aesthetics, and a small VLM caption (BLIP-2 / Florence-2-class).
- **dedup**: Hamming-distance prefilter on top-16 bits of pHash → bucket;
  embedding-based confirm with a CLIP/V-JEPA features (planned v1).
- **curate**: rules + learned filter (see §7) → shards.

## 7. Adversarial open questions (assigned to next sessions)

1. **Egocentric license-clean gap.** EPIC-KITCHENS et al. are
   research-license-only. Can we partner with home cooks via an opt-in
   community, e.g. a PeerTube instance, to bootstrap a CC0 egocentric
   cooking corpus? Cost/feasibility note in `docs/research/`.
2. **World-model utility score.** Train a small predictor whose label is
   "did including this clip in pretraining improve next-frame loss on a
   held-out probe?" Use as a learned filter à la DCLM's fastText filter.
3. **Cuisine and demographic bias audit.** HowTo100M is heavily
   English/Western. Our license-clean sources (Wikimedia, Internet
   Archive) are likely *worse* on this axis. Plan an audit: cuisine,
   language, gender of presenter, kitchen geography.
4. **Eval-set contamination.** When (not if) we benchmark against
   YouCook2 / EPIC-KITCHENS, we must ensure no shared video IDs or
   near-duplicates leak in. Build a "blocklist" pHash set from the
   eval-set keyframes and reject any candidate within Hamming 8.
5. **PII / face / address handling.** Home kitchens leak addresses (mail
   on a counter, a window view). Plan: face-blur and OCR-redaction by
   default before any redistribution.
6. **Licence ambiguity at the long tail.** Wikimedia-canonicalized
   uploads sometimes carry conflicting tags (file is CC-BY but
   description page lists "non-commercial only"). Plan: per-source
   adapter must emit both the license string and a *confidence*; below
   confidence threshold ⇒ exclude.
