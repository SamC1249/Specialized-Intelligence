Adversarial verdict on day-one work: **the scaffolding is sound, but
the curation theory is thin.** Concretely, three gaps I am now treating
as today's mandate:

1. **No eval-set decontamination.** Day-one §7.4 promised a pHash
   blocklist; nothing in the pipeline actually consumes one. This is
   the single highest-leverage gap, because if we *ever* benchmark on
   YouCook2, EPIC-KITCHENS, HD-EPIC, or any other research-license eval
   set, leakage from training is a publication-bar failure.
2. **No "what does good curation look like?" theory of the case.** The
   dry-run treats all kept clips as equal. The 2025–2026 literature
   (V-JEPA 2, V-JEPA 2.1, Summer-22B) is unanimous: *cluster-based
   retrieval against a curated target distribution* is the dominant
   approach. We have not even written down what target distribution we
   are aiming at.
3. **No "license-confidence" contract for adapters.** Day-one §7.6
   flagged that Wikimedia long-tail uploads carry conflicting tags.
   Without a typed `license_confidence` field per row, any threshold
   filter we eventually write is operating on a contract that doesn't
   exist.

## 2. What I shipped today

### 2.1 Eval-set decontamination is now a first-class pipeline step

- `components/eval_blocklist.py`: `load_blocklist()` reads a `.jsonl`
  file of `{phash64, source}` rows; `BlockList(threshold)` answers
  `contains` / `nearest`; `apply(clips, blocklist)` marks colliding
  clips `excluded=True` with `exclusion_reason="eval_blocklist_hit
  (d=<n>, ref=<phash>)"`.
- `scripts/pipeline_dry_run.py` now takes optional
  `--eval-blocklist <path>`; on hit, the relevant clip is excluded
  *after* intra-pool dedup, so the failure mode is recorded, not
  silently dropped.
- Two e2e tests prove the path: (a) seeding the blocklist with the
  exact pHashes the pipeline emits drives every clip to excluded and
  the shard's `clip_ids` to `[]`; (b) an unrelated hash leaves the
  shard untouched. Six unit tests cover the loader and the
  threshold/edge behaviour of `BlockList`.

This required relaxing `SHARD_SCHEMA.clip_ids` from `minItems: 1` →
`0`, and an explicit clause in `db_structured.md` §2.3: *an empty
shard is the canonical signal that the run produced zero shippable
clips for this purpose.* I prefer this to silently emitting no shard,
because downstream pipelines can now distinguish "we ran and rejected
everything" from "we never ran." Pipeline version bumped 0.0.1 → 0.0.2.

### 2.2 Adapter license-confidence enforced via pre-commit

- `scripts/precommit_require_license_field.py`: any `.py` file under
  `components/sources/` that mentions the literal `license` must also
  expose `license_confidence`. Five unit tests over happy / failing /
  ignored cases.
- `.pre-commit-config.yaml` and `.github/workflows/ci.yml` wired up.
  Hook is scoped to `^components/sources/.*\.py$`; non-adapter files
  are not affected.
- CI now runs unit tests on a matrix of Python 3.11 and 3.12 (was

## 3. The theory of curation, made explicit

Based on V-JEPA 2 + DataComp + our day-one priors, the *full* curation
DAG we're heading toward looks like:

```
                        ┌──────────────────────────┐
target_distribution ─┐  │ license-clean seed pool  │
(curated CC seeds)   │  │ (Wikimedia + IA + CC     │
                     ▼  │  PeerTube cooking subset)│
                ┌────────┐  │
candidate pool ─►│cluster │──► resampled candidates
(CC video       │retrieval│
 from CC + IA + │+ filter │
 PeerTube +     └────────┘
 CC search)
                     │
                     ▼
              dedup (pHash → embedding)
                     │
                     ▼
         eval_blocklist (today's deliverable)
                     │
                     ▼
        NSFW + face/OCR redaction (planned)
                     │
                     ▼
                  shard
```

Today's shipped step (`eval_blocklist`) is the *gating* step before
shards leave the pipeline. The cluster-retrieval block is the *next*
critical step, planned for the next session.

## 4. Decisions, recorded

- **D-2026-06-24-1**: An empty shard (`clip_ids: []`) is valid and
  signals "we ran and rejected everything for this purpose." Replaces
  the day-one assumption that shards are minItems=1.
- **D-2026-06-24-2**: Eval-set decontamination is *not optional* in v1+.
  v0 pipeline supports it via `--eval-blocklist`; v1+ will require a
  non-empty blocklist for shard purposes `eval` / `probe` / `held_out`.
- **D-2026-06-24-3**: Every adapter under `components/sources/` must
  emit a numeric `license_confidence ∈ [0, 1]`. Threshold to keep is a
  per-purpose parameter (default 0.8 for `pretrain` shards). Enforced
  by pre-commit and CI.
- **D-2026-06-24-4**: V-JEPA-2-style cluster retrieval is the *target
  architecture* for filtering, with a license-clean seed distribution
  composed of (a) Wikimedia cooking categories, (b) Internet Archive
  Prelinger home-economics films, (c) curated PeerTube CC0/CC-BY/CC-BY-SA
  cooking videos. We do **not** seed with HD-EPIC / EPIC-KITCHENS /
  Ego4D — they're CC-BY-NC and would launder a non-commercial bias into
  the pipeline.

## 5. Adversarial open questions (carry forward to next session)

1. **Seed pool sufficiency.** Is a seed pool of (Wikimedia + IA +
   curated PeerTube cooking) large enough to drive cluster retrieval to
   pull useful candidates from Common Crawl? Day-one §7.1 still open.
   *Method to test: build the seed pool, compute embeddings on
   keyframes, measure intra-pool entropy + diversity against a
   reference (e.g. an HD-EPIC keyframe sample).*
2. **License-confidence calibration.** Are uploader-set tags on
   Wikimedia/IA/PeerTube reliable enough that `license_confidence` can
   default to 0.9, or do we need a per-source default (e.g.
   Wikimedia 0.95, PeerTube 0.7, IA 0.6 pending a second-pass auditor
   step)?
3. **Embedding model for retrieval.** CLIP filters amplify English /
   Western bias (DataComp critique). V-JEPA / DINO-v2 embeddings are
   more action / motion-aware. *Hypothesis: a V-JEPA-2 ViT-L encoder
   on keyframes outperforms CLIP-on-keyframes for retrieval into our
   cooking target distribution; held-open question for an empirical
   ablation when we have GPU budget.*
4. **Carry-forward from day-one §7.5 (PII / face / address).** Not
   shipped today; deliberately deferred until we are within a session
   of actually shipping frames.
5. **Carry-forward from day-one §7.3 (cuisine / demographic audit).**
   The DataComp note today reinforces this; still open.
